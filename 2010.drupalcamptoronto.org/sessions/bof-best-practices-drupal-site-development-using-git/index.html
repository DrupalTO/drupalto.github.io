BOF: Best Practices for Drupal site development using Git

Submitted by sensespidey on Thu, 2010-10-07 15:02
Code Development and Performance
Intermediate

Recently the Drupal community has agreed to switch our official Revision Control System from the venerable and ancient CVS to the powerful and newfangled Git. There are many issues associated with migrating the Drupal core and contrib repositories to Git, and these are being tackled with a formidable team of volunteers from the community at large.

Having introduced myself to Git, I am convinced it is a better system, but I have yet to figure out the best way to go about site development using Git. Typically, I've kept all of core in a different branch of the same SVN/CVS repository as my sites, check them out separately, and use symlinks trickery to pull everything together. In the new paradigm, this doesn't make much sense.

Does one keep a repository with all of core + the sites/ dir with any contrib/custom modules in it? Keep one repository for core and one just for the site itself? Leave out core altogether, and just use a drush make file to pull it in directly from drupal.org? I have been moving in the direction of keeping a repository of "platform" makefiles which know how to build a particular set of core + contrib modules/features for a site, and then keeping a repository per-site which contains the entire sites/ directory and any custom/contrib code that is specific to the site. I'm interested in discussing and sharing ideas around this new tool and how people are using it to best effect.

DISCLAIMER: I am by no means an expert on the mechanics of git, so aside from a very high-level intro and perhaps some basic workflow pointers and resources, this session will NOT be an introduction to using Git itself. Instead, the focus is on how to apply the git paradigm (distributed, small repositories) to Drupal site development.
